# 教师课程交付产品总任务

更新时间：2026-07-17

## 总目标

把产品的生成单位从“一份 PPT”升级为“一节可直接执行的课”。

老师确认的核心对象必须是课堂方案，而不是固定页数的目录。系统在课堂方案确认后，再派生学生投影课件、教师讲稿、答案、板书、作业和可持续修改的版本。

## 当前根因

1. 策划层把课堂事件与 PPT 页面一一对应，课堂结构被页面结构反向限制。
2. 页数仍主要由课时长度映射，未由教学任务、内容密度和呈现节奏共同推导。
3. 输入合同缺少学生基础、常见困难、课堂设备和评价侧重等关键教学约束。
4. 教师确认页面编辑的是“第几页讲什么”，不是“这节课为什么这样上”。
5. 学科差异主要停留在文案和少量内容分支，尚未形成物理实验探究、语文精读等稳定的教学策略合同。
6. 教师包虽然已生成，但尚未成为主流程中的可见、可修改和可下载交付物。
7. 工程成功、样品可看和老师可直接上课尚未由同一套验收标准约束。

## 目标产品链

```text
教师输入教材、课题、学情和课堂约束
-> 系统生成课堂蓝图
-> 教师确认目标、重难点、课堂主线和时间分配
-> 系统按课堂事件派生动态页面
-> 同版本生成学生课件与教师交付包
-> 教师局部修改并保存不可变版本
-> 导出、重开、再次下载
-> 真实教师过课并回写问题
```

## 分阶段任务

### P0-1 课堂蓝图合同

- 独立定义 LessonBlueprint，不再用 slidePlan 代替教学策划。
- 明确课型、驱动问题、学生基础、重难点、学习证据、课堂约束和教师待确认项。
- LessonEvent 可以关联零页、一页或多页，禁止默认事件数等于页数。
- 完成标准：物理《楞次定律》和语文《背影》产生不同课型、不同课堂主线和准确的 45 分钟预算。

### P0-2 教师输入合同

- 增加学生基础、常见困难、班级规模、可用设备和评价侧重。
- 允许信息未知，但必须显式标记为假设并要求教师确认。
- 完成标准：策划结果能解释每项关键决策基于教师输入还是系统假设。

### P0-3 课堂方案确认界面

- 先展示课程定位、目标、重难点、课堂主线和时间分配。
- 页面目录降为派生结果，教师先改课堂事件，再改页面。
- 完成标准：老师不进入画布也能判断这节课是否符合自己的班级。

### P0-4 学科教学策略

- 物理：现象观察、变量变化、证据记录、规律建构、方向判断、纠错迁移。
- 语文：整体感知、文本证据、语言品读、情感推进、朗读表达、迁移写作。
- 完成标准：不能仅替换学科名得到另一学科的课堂流程。

### P0-5 交付界面

- 同时提供学生投影版、教师讲稿、答案、板书和作业。
- 教师备注按页面覆盖，课堂事件按教学过程组织，两者保持引用关系但不强制一一对应。
- 完成标准：老师能从主界面查看并修改全部交付物。

### P0-6 两条黄金链路

- 高二物理《楞次定律》。
- 初二语文《背影》。
- 完成生成、确认、修改、保存、导出、重开、再次下载和真实 WPS/PowerPoint 抽检。
- 完成标准：两位对应学科教师能按教师包完成一次 45 分钟试讲并给出可记录反馈。

## 不接受的伪完成

- 不能用“生成了 9 页”证明课堂完整。
- 不能用 API 返回 200 证明老师能直接上课。
- 不能用本机可打开证明客户电脑不乱码。
- 不能用教师备注字段存在证明备注有实际教学价值。
- 不能用两个手工特例证明学科矩阵成熟。
- 不能在没有真实教师试讲时标记为商业可交付。

## 当前执行顺序

1. 完成课堂蓝图合同与验证器。
2. 让页面计划由课堂蓝图派生，并解除事件数等于页数的假设。
3. 改造教师确认界面。
4. 完成两门课浏览器黄金链。
5. 进入真实教师与目标办公软件验收。
