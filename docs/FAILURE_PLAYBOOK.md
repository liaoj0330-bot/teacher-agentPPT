# Teacher AgentPPT 卡点复盘与防复发手册

## 核心教训

项目过去最大的成本不是“某个功能写不出来”，而是以下三类错位：

1. 版面确定了，但底层交互和后端事实链没有理顺；
2. 自动化或局部接口通过了，但真实用户流程没有完整运行；
3. 报告只写进度，没有记录失败模式，下一次又从头踩坑。

## 主要问题与处理规则

### 1. 新旧入口混用

- 表象：多个教师 PPT 页面都能打开，功能看似相似。
- 根因：没有唯一主入口，旧交互和新后端互相污染。
- 处理：先确认 `/teacher-ai-ppt`，旧能力只迁移领域模块和 API，不复制旧页面状态。
- 防复发：入口验收必须检查 URL、页面身份、请求路径和版本写入对象。

### 2. 只修单个 PPT，不修生成逻辑

- 表象：某份成品被修漂亮了。
- 根因：只改输出，没有倒推策划、内容、布局和交互。
- 处理：所有成品问题必须归因到 ContentPlan、DeckSpec、LayoutContract、RenderScene 或导出适配器。
- 防复发：同一修复至少用两个不同课题回归。

### 3. 固定 9 页冒充规划

- 表象：每次都能快速生成完整课件。
- 根因：固定模板替代课程策划。
- 处理：由状态机根据课程目标和教学环节决定页数与页面职责。
- 防复发：回归必须包含不同页数，并验证每页 role 不重复堆砌。

### 4. 前端能点但没有真实写入

- 表象：按钮、Toast 和页面变化都存在。
- 根因：只修改浏览器内存，没有服务器版本和可查询记录。
- 处理：每次修改携带 `baseVersionId`，成功后创建新 CoursewareVersion。
- 防复发：读取旧版本确认不变；过期基线必须返回 409。

### 5. 浏览器和 PPTX 两套渲染逻辑

- 表象：网页很好看，导出的 PPTX 完全不同。
- 根因：预览和导出分别拼布局。
- 处理：两者消费同一 RenderScene。
- 防复发：比较 sceneId/layoutId/pageCount，并解包 PPTX 验证独立对象。

### 6. 测试显示通过，但断言没有执行

- 表象：脚本退出码为 0 或旧报告显示 10/10。
- 根因：补丁应用失败后命令继续；断言被插到 `process.exit` 之后；测试读取旧报告。
- 处理：应用失败立即停止；报告每步落盘；断言位置和时间戳必须检查。
- 防复发：测试报告记录源码提交、开始时间、结束时间和实际断言数量。

### 7. 响应 Promise 先创建，按钮点击失败

- 表象：脚本等待接口数分钟，像是模型或后端很慢。
- 根因：按钮点击失败后留下未处理的 `waitForResponse`。
- 处理：先等待按钮可见，再用 `Promise.all([waitForResponse, click])`。
- 防复发：生成前检查浏览器动作名、可见状态和服务器请求日志。

### 8. 错怪火山或品川出图

- 表象：生成流程卡住，直觉认为图片模型不行。
- 根因：请求根本没有进入 `/api/generate-image`。
- 处理：按“按钮 → generate-ppt → generate-image → 供应商 HTTP → 图片字节”逐层定位。
- 防复发：供应商失败必须有真实请求 ID、HTTP 状态或超时证据。

### 9. 3002 有进程但服务不可用

- 表象：端口被占用，启动新服务报 `EADDRINUSE`。
- 根因：遗留 Node 进程仍监听但 HTTP 无响应。
- 处理：核对 PID、启动时间和路径；只停止已确认的旧进程；重启后检查 HTTP 200。
- 防复发：会话交接记录 PID 和健康状态；启动前先探测端口。

### 10. 旧环境掩盖冷启动缺陷

- 表象：原工作区 `tsc` 通过，新仓库安装后出现 Prisma 隐式 any。
- 根因：旧 `node_modules` 已生成 Prisma Client，新安装没有自动生成。
- 处理：增加 `postinstall: prisma generate`，在干净目录重新安装和构建。
- 防复发：发布验收必须在没有 node_modules 的目录执行。

### 11. Git 发布夹带数据库或工作树指针

- 表象：看似创建了新仓库，实际连接回原 worktree；`prisma/dev.db` 被暂存。
- 根因：Windows worktree 的 `.git` 是文本指针，复制目录时被带入。
- 处理：发布目录明确排除 `.git` 文件和目录；风险文件扫描通过后再提交。
- 防复发：`git rev-parse --git-dir`、暂存风险扫描和密钥扫描是推送前强制门。

### 12. 版面先确认、产品链后补

- 表象：视觉版面看似完成，但后端、编辑、导出和验收混乱。
- 根因：把设计确认当成产品完成。
- 处理：版面只算 Visual Contract；必须继续验证交互、事实源、版本、导出和失败恢复。
- 防复发：汇报中分别列“版面确认、工程通过、真人验收、商业采用”。

### 13. 分钟级图片任务放在同步请求里

- 表象：页面一直生成、浏览器等待数分钟、用户只看到无响应。
- 根因：单页 1–2 分钟的外部任务被塞进单个 HTTP 请求或浏览器动作。
- 处理：提交后立即返回 `taskId`；后台最多 2 路受控并发，页面完成即持久化。
- 防复发：中断浏览器后任务仍可查询和继续，已完成页面不能丢失。

### 14. 本地降级冒充远程成功

- 表象：HTTP 200、页面有图片，但无法确认是否真的调用指定供应商。
- 根因：远程成功、本地 fallback 和失败共用同一成功状态。
- 处理：统一记录 `remote_success`、`fallback`、`failed`，并保存 provider 与 requestId。
- 防复发：真实模式可禁止 fallback；供应商验收必须检查图片字节与版本资产。

### 15. 一页失败导致整套重跑

- 表象：第八页失败后，前七页再次调用图片模型。
- 根因：没有页面级状态、幂等键和已完成资产复用。
- 处理：使用 `versionId + slideId + promptHash` 标识任务，只重试失败页。
- 防复发：故障注入后断点续跑，已成功页面的供应商调用次数必须保持不变。

### 16. 替代 PPTX 冒充产品闭环

- 表象：网页流程失败后，本地脚本生成一份 PPTX，于是汇报为系统已经完成。
- 根因：文件交付与产品真实路径没有分层。
- 处理：本地产物只作为内容/版式样例；产品验收必须经过主入口、版本链、RenderScene 和 Artifact。
- 防复发：所有产物记录 `sourcePath` 和 `acceptanceScope`。

### 17. 只解析 PPTX，不做视觉验收

- 表象：文件能打开、页数正确、图片尺寸统一，但仍可能有空白、溢出或错图。
- 根因：结构检查代替逐页渲染检查。
- 处理：导出后自动渲染逐页 PNG，检查空白、溢出、重叠和冻结版本一致性。
- 防复发：没有渲染报告时只能标记 `verification`。

## 每次遇到新问题的固定提问

1. 用户实际点击了什么？
2. 浏览器发出了哪个请求？
3. 后端进入了哪个处理分支？
4. 哪个事实源发生了变化？
5. 新旧版本是否都可查询？
6. 浏览器和导出是否读取同一事实？
7. 失败证据来自本轮还是旧报告？
8. 修复是否经过不同课题回归？
9. 人类最终看到了什么、能否继续操作？
