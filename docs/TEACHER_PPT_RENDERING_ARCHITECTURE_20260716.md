# 教师 PPT 助手渲染架构评估与升级路线

日期：2026-07-16

## 结论

当前系统的强项是内容策划、版本冻结、图片生成、原生 PPTX 导出和基础几何检查；短板集中在“渲染决策层”。原系统有版式名称和槽位，但缺少成熟演示产品常见的构图家族、主区域、文本流、密度预算、字体预算、视觉策略和模板继承闭环，因此最终产物容易退化为固定的左右分栏。

本轮不替换 PptxGenJS。它继续负责原生 OOXML 生成。新增的核心是位于内容与 PptxGenJS 之间的 Scene Composition 层，使浏览器预览、课堂预览、缩略图、评分和 PPTX 导出消费同一份场景真相。

## 成熟 PPT 产品做深的维度

1. 沟通任务：每套演示先定义受众、课堂结果、核心主线和结束动作。
2. 叙事结构：页面不是目录堆叠，而是问题、观察、解释、应用、反馈的递进关系。
3. 页面角色：封面、目标、情境、概念、例题、练习、纠错、总结各有不同任务。
4. 构图家族：同一主题至少具有封面主视觉、左图右文、右图左文、三段任务、对比舞台、步骤推导等不同轮廓。
5. 主区域：先定义标题带、证据区、视觉区、任务区、反馈区，再放具体对象。
6. 文本流：明确文本框的阅读顺序、分组和跨槽位流动，不按数组顺序随意均分。
7. 密度预算：每种布局限制字数、内容块和图片数量，超预算时合并、换布局或拆页。
8. 字体预算：标题、正文、注释有最低字号；禁止依赖 PowerPoint 动态 shrink 兜底。
9. 视觉策略：图片承担情境、证据或记忆锚点，不与正文争夺同一区域。
10. 数据对象：表格、图表、公式、坐标系、流程图必须是语义对象，不是装饰图片。
11. 模板继承：支持母版、版式、命名形状、占位符和主题字体，而不只是读取模板颜色。
12. 编辑性：核心文本、表格、图表和教学对象保持为原生对象，并能在前端定位到对应槽位。
13. 预览一致性：编辑器、缩略图、课堂播放和最终 PPTX 使用同一 Scene。
14. 质量闸门：内容、几何、字体、图片、OOXML、真实渲染截图和教学正确性分别留证。
15. 反馈闭环：将教师修改、失败页面和人工复核结果反馈到布局选择与提示词策略。

## GitHub 与成熟 Skill 决策

### 继续采用：PptxGenJS

- 仓库：gitbrent/PptxGenJS
- MIT，TypeScript，约 5.8k stars，当前 npm 版本 4.0.1。
- 适合从 Scene 创建原生文本、图片、表格、图表、形状和母版。
- 不负责内容策划、布局选择、浏览器预览和真实渲染验证，这些必须由本系统提供。

### 计划采用：pptx-automizer

- 仓库：singerla/pptx-automizer
- MIT，TypeScript，2026-06 仍在维护。
- 适合导入专业 PPTX 模板、保留母版/版式、按命名形状修改文本图片表格图表，并能用 PptxGenJS 补充新对象。
- 限制：仅服务端；复杂 layout 中的图片/图表有限制；动画不在范围内。
- 接入前置条件：上传模板字节持久化、模板所有权、模板版本、命名形状清单和回退策略。当前接口只返回 manifest，因此暂不直接切换主导出路径。

### 已嫁接逻辑：Codex Grid Presentation Skill

采用其 26 布局注册表中的合同设计：templateUse、layoutFamily、contentRoles、slots、majorRegions、textFlows、densityBudget、typographyBudget 和 assetSlots。没有复制视觉皮肤，而是转换成教师课件的 9 类 Scene Composition。

### 不采用为 PPTX 主路径

- Slidev：适合开发者 HTML 演示和交互，不以原生可编辑 PPTX 为核心。
- Marp：适合 Markdown 批量演示，布局自由度和对象级编辑不足。
- Reveal.js：成熟 HTML 演示框架，但不是 OOXML 编辑器。
- python-pptx：成熟，但与现有 TypeScript/Next/PptxGenJS 双栈重叠，且不提供渲染器。

## 目标架构

1. Content Layer：教师任务、教材证据、教学目标、内容块和讲稿。
2. Planning Layer：页面角色、课堂问题、必须证明、视觉形式和信息密度。
3. Composition Layer：构图配方、主区域、文本流、视觉策略、密度与字体预算。
4. Scene Layer：统一的文本、图片、表格、图表、公式、形状和锚点对象。
5. Adapter Layer：Browser、PptxGenJS、未来 pptx-automizer 三个适配器。
6. Artifact QA：图片清单、几何、字体、OOXML、截图、OCR/视觉语义和人工复核。
7. Product Feedback：前端换版、单页重试、教师审核和失败案例回流。

## 本轮已实施

- 增加 9 类教师 Scene Composition。
- 每个配方包含 use/avoid、主区域、文本流、密度预算、字体预算、视觉策略和独立配色。
- 旧版式不再统一回落成单一左右分栏。
- Browser 与 PPTX 读取同一构图颜色和装饰元素。
- 主编辑画布、缩略图和课堂预览加载真实生成图片。
- 设计面板显示当前构图、密度、字号和字数预算。
- 手动更换版式时清除旧 selectedLayout 锁定，允许重新选择构图。
- Image2 提示词按目标、情境、概念、例题、练习、纠错、总结分流。
- 例题与练习图片禁止生成数字和算式，数学表达由原生 PPT 对象承担。

## 验收标准

- 连续 6 页不得使用同一构图家族。
- 每页标题、正文、图片和结构化对象必须绑定 Scene 槽位。
- 正文不得低于 16pt；标题不得依赖自动 shrink。
- 图片与文字重叠为阻断错误。
- 图片清单必须是可解析位图并满足分辨率要求。
- 预览和 PPTX 的元素数量、锚点和主区域一致。
- OOXML 可编辑覆盖率不低于 90%。
- 正式交付必须有真实渲染截图；没有截图只能是人工复核稿。
- 教学图片中的数量、关系和题目必须通过人工或视觉模型复核。
