# Teacher AgentPPT 管理员与 100 人内测操作手册

## 1. 角色分工

小团队可以一人兼任多个角色，但每个角色必须写下真实姓名和联系方式：

- 内测负责人：决定发码、暂停、恢复和扩批。
- 运营值班：维护老师名单、支持群、反馈和日报。
- 工程值班：处理日志、发布、回退和数据恢复。
- 内容审核：处理教材、知识、答案和课堂设计问题。
- 隐私负责人：处理材料、账号、密钥和删除请求。

## 2. 上线前一天

1. 确认 GitHub 的目标发布提交号。
2. 完成数据库和 `artifacts/` 备份及恢复演练。
3. 在服务器配置文本、图片、邀请制和管理员环境变量。
4. 用内部账号走完注册、生成、图片、导出、重下载和反馈。
5. 在真实 WPS 和 PowerPoint 各打开至少一份 PPTX。
6. 访问管理员看板，确认指标能读取。
7. 建立 10 位种子老师名单和支持群。
8. 明确 08:00-22:00 值班人和紧急联系方式。

任何硬门槛未通过，都只修复和复测，不发首批邀请码。

## 3. 100 个名额怎么发

按 `10 + 20 + 30 + 40` 四批发放，每批至少观察一个完整工作日。不要一次把 100 个名额全部放出。

当前代码的 `BETA_MASTER_INVITE_CODES` 是共享主码列表，不能自动做到一码一人、激活失效和撤销审计。在数据库一次性邀请码完成前：

- 每个老师必须人工登记邮箱、学科、年级、邀请码、发放时间和激活状态。
- 邀请码只能私发，不发群公告，不放截图，不写入 GitHub。
- 发现共享码扩散时，立即从服务器环境变量移除并重启服务。
- 100 人公开扩散必须等数据库级一次性邀请码完成。

## 4. 500 积分方案

新注册老师自动领取 500 积分。当前标准成本：生成课件 24、首次导出 8、成功图片每张 6、AI 精修每页 4。

管理员补额接口为 `POST /api/admin/credits`，只允许 `BETA_ADMIN_EMAILS` 中已经登录的账号调用。请求格式：

```json
{
  "emails": ["teacher@example.com"],
  "amount": 100,
  "reason": "内测故障补偿"
}
```

每次补额必须记录批次 ID、邮箱、数量、原因和操作人。不要因为老师反馈失败就先批量补分；先核对积分账本是否确实发生了错误扣费。

## 5. 每天的固定动作

### 08:00

- 检查夜间错误、开放 P0/P1、队列和备份结果。
- 检查文本和图片健康配置。

### 09:00

- 根据前一批指标决定当天是否发码。
- 未达到扩批门槛时保持原批次，不用“多发一点再看看”。

### 12:00

- 第一次反馈分诊，合并重复问题。
- 教材和答案问题转内容审核；失败和性能问题转工程值班。

### 17:00

- 检查生成、导出、图片成功率，P50/P90 延迟和最老任务等待时间。
- 检查图片调用和积分消耗是否异常。

### 20:30

- 输出日报：发码数、激活数、生成用户、导出用户、失败环节、P0/P1、反馈 SLA、次日动作。

### 22:00

- 停止非紧急发布；P0 继续处理。

## 6. 管理员看板怎么读

访问 `/teacher-ai-ppt/admin`。只有 `BETA_ADMIN_EMAILS` 中的登录邮箱可以访问。

重点看：

- 使用率：注册、活跃、生成、导出、图片页数和积分消耗。
- 稳定性：生成完成率、导出成功率、图片成功率、P50/P90 延迟。
- 队列：排队任务和最老等待分钟数。
- 反馈：开放 P0、P1 和全部未关闭反馈。

看板绿色只表示当前未触发硬停，不代表可以跳过观察期和人工签字。

## 7. 扩批门槛

同时满足才进入下一批：

- 没有未关闭 P0，P1 不超过 2 个。
- 生成完成率至少 95%。
- 导出成功率至少 98%。
- PPTX 重复下载哈希一致率 100%。
- 没有教材错配、答案错误或乱码集中爆发。
- 队列没有连续 30 分钟增长。
- P0/P1 反馈在 SLA 内被确认和分派。

样本少于 5 个时，不因百分比波动自动判定健康；隐私、密钥、数据丢失和损坏文件不受样本量限制。

## 8. 立即停止发码的情况

- 隐私或 API Key 泄露。
- 数据丢失或账号串数据。
- 下载的 PPTX 批量损坏。
- 30% 以上活跃老师无法生成或导出。
- 多位老师教材严重错配或核心答案错误。
- 队列持续增长且没有可用绕行方案。

停止发码不等于删除数据。保留失败任务、请求 ID、版本、时间窗和指标快照。

## 9. 消防队启动

按以下格式向消防队 Agent 提交：

```text
/fire report 发生时间；页面或接口；影响人数；错误提示；反馈ID；项目/版本ID；最近发布提交
```

常用命令：

- `/fire status <事故ID>`：查看影响、动作和下一次更新时间。
- `/fire contain <事故ID>`：执行或建议可逆止损。
- `/fire recover <事故ID>`：按恢复门槛验收。
- `/fire close <事故ID>`：负责人签字后关闭。

完整指令见 `agents/private-beta-fire-response-agent.md`，机器策略见 `project-state/teacher-agentppt.fire-response-agent.json`。

## 10. 图片故障处置

1. 将 `BETA_IMAGE_GENERATION_ENABLED=false`。
2. 重启应用，确认无图课件仍能生成和导出。
3. 停止整套图片自动重试，只保留失败页证据。
4. 核对成功页扣分、失败页不扣分。
5. 供应商恢复后先用内部账号单并发测试，再恢复 5 位老师。

## 11. 生成空响应或 JSON 错误

1. 暂停新增生成，但保留登录、项目、下载和反馈入口。
2. 保存请求 ID、错误文本、发生时间、模型、网关和发布提交。
3. 检查上游超时、响应截断、内存和解析日志。
4. 确认失败任务没有扣生成积分。
5. 修复后完成两次内部生成，再开放 5 位老师观察。

## 12. PPTX 损坏或乱码

1. 暂停新导出，保留项目和版本。
2. 保存损坏文件、产物 ID、版本 ID 和哈希。
3. 比较服务器产物与重复下载文件哈希。
4. 在 WPS 和 PowerPoint 分别打开抽样。
5. 修复后验证中文、英文、公式、字体、图片和可编辑对象。

## 13. 对老师怎么通知

故障通知只说四件事：受影响功能、可用绕行方法、下一次更新时间和数据是否安全。不要把内部日志、完整请求、账号信息或密钥发到群里。

示例：

```text
当前图片生成功能暂时不可用，无图课件生成和 PPTX 导出仍可正常使用。已经暂停图片入口，失败图片不会扣积分。我们将在 30 分钟后更新处理进展。
```

恢复通知必须在内部账号、5 位老师和监控窗口均通过后发布。

## 14. 当前未完成项

- 数据库级 100 个一次性邀请码和批次台账。
- 服务端每日课件/图片硬配额。
- PostgreSQL、对象存储和独立任务队列。
- 云主机、网关和供应商自动告警。
- 管理员反馈工单队列和更细 RBAC。
- 自动备份、清理和恢复演练任务。

这些项目没有完成前，可以做小批邀请制内测，但不能宣称支持 1000 人公开自助或多实例商业化上线。
