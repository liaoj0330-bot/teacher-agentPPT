# Teacher AgentPPT 自媒体复盘素材库

本文件把真实开发代价转化为可公开表达的内容素材，但不泄露密钥、私人教材、内部账号或未经确认的商业数据。

## 选题 1：版面确定，为什么产品仍然不能用

- 钩子：一套 PPT 界面看起来已经完成，为什么背后的产品可能还是乱的？
- 冲突：视觉确认被误当成产品完成。
- 真实教训：交互、版本、导出和失败恢复没有同一事实源。
- 核心观点：版面是视觉合同，不是完整产品。
- 可视化：版面层 → 交互层 → 版本层 → 导出层 → 验收层。

## 选题 2：10/10 测试通过，为什么用户仍然点不通

- 钩子：后端测试全绿，用户为什么还会卡在生成按钮前？
- 冲突：局部测试与真实用户旅程不一致。
- 真实教训：自动化断言可能没有执行，或者按钮根本没有发出请求。
- 核心观点：测试通过必须说明测试了哪一层。
- 可视化：单元测试、API E2E、浏览器按钮、真人验收四层金字塔。

## 选题 3：不要急着怪大模型

- 钩子：生成失败，真的是火山或图片模型不行吗？
- 冲突：流程卡住时，人会优先归因最复杂的外部模型。
- 真实教训：当时 `/api/generate-image` 根本没有被调用。
- 核心观点：先沿请求链取证，再判断供应商。
- 可视化：点击 → 生成 API → 图片 API → 供应商 → 图片字节。

## 选题 4：从固定 9 页到真正的课件策划

- 钩子：为什么“每次稳定生成 9 页”反而可能是假成熟？
- 冲突：固定模板很稳定，但无法反映真实教学结构。
- 真实教训：页数、例题、练习和反馈必须由课程目标决定。
- 核心观点：成熟产品不是固定页数，而是稳定地产生合理规划。
- 可视化：Teacher Context → State Machine → DeckSpec → Pages。

## 选题 5：为什么 AI 产品必须保留不可变版本

- 钩子：AI 帮你修改 PPT，为什么不能直接覆盖？
- 冲突：覆盖最省事，但无法追责、回退和验证。
- 真实教训：旧版本不变、过期编辑 409、导出绑定版本哈希。
- 核心观点：可追溯比“看起来智能”更重要。
- 可视化：V1 → V2 → V3 分支与 Artifact 绑定。

## 选题 6：我为“假完成”付出的代价

- 钩子：最贵的不是代码写错，而是你以为它已经完成。
- 冲突：截图漂亮、按钮存在、退出码为 0，都可能是假成功。
- 真实教训：必须检查请求、版本、产物结构和真人结果。
- 核心观点：任何完成都需要实现、执行、证据和人工验收四件套。
- 可视化：四道完成门。

## 内容生产规则

- 只使用已经验证的事实。
- 不把工程进度百分比写成商业成熟度。
- 不公开真实密钥、教师私人材料、测试账号和本机路径。
- 讲清楚“当时为什么会误判”，而不只展示最终成功。
- 每篇内容保留一个可执行方法，让读者能复用。
